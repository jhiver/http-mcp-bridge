Introduction

SaraMCP is a powerful platform that transforms HTTP APIs into MCP (Model Context Protocol) tools that can be used by Claude and other LLMs. This tutorial will walk you through the complete process of creating, configuring, and using MCP servers.

What You'll Learn

  • How to create reusable toolkits of HTTP-based tools
  • How to configure MCP servers with tool instances
  • How to manage parameter bindings with typed variables
  • How to test and execute MCP protocol operations
  • How to connect your MCP server to Claude or other LLMs

Getting Started

Step 1: Access SaraMCP

Navigate to https://saramcp.com to access the homepage. You'll see a clean interface with options to log in or sign up.

Step 2: Log In or Sign Up

Enter your credentials to access the platform. After logging in, you'll be taken to the dashboard.

The dashboard shows your statistics including:

  • Number of servers, toolkits, tools, and instances
  • Execution statistics (24h, 7d, all-time)
  • Server performance metrics

Creating Your First Toolkit

What is a Toolkit? A collection of related HTTP API tools that can be reused across multiple servers.

Step 3: Navigate to Toolkits

Click "My Toolkits" in the sidebar to see your existing toolkits.

Step 4: Create a New Toolkit

Click the "+ CREATE TOOLKIT" button to open the creation form.

Fill in the details:

  • Toolkit Name: "Public APIs Toolkit"
  • Description: A collection of free public APIs
  • Visibility: Choose "Public" to allow others to discover and use your toolkit

Step 5: Add Tools to Your Toolkit

After creating the toolkit, you'll see the toolkit details page. Click "+ CREATE TOOL" to add your first tool.

Step 6: Configure Your First Tool

Let's add the Chuck Norris Jokes API:

  • Tool Name: Chuck Norris Jokes
  • Description: Get a random Chuck Norris joke
  • HTTP Method: GET
  • URL: https://api.chucknorris.io/jokes/random
  • Timeout: 30000ms

Step 7: Add More Tools

Repeat the process to add more tools. We added:

  1. Chuck Norris Jokes - Simple GET request with no parameters
  2. Currency Exchange Rates - GET with {{string:base_currency}} parameter
  3. Advice Generator - GET with optional {{string:query}} parameter

Understanding Parameters

SaraMCP supports typed parameters in your API templates:

Syntax Type Description
{{string:name}} String String parameter
{{integer:id}} Integer Integer parameter
{{number:rate}} Number Decimal number
{{boolean:debug}} Boolean Boolean value
{{json:data}} JSON JSON object/array
{{url:endpoint}} URL Validated URL

Creating and Configuring an MCP Server

Step 8: Create a New Server

Navigate to "My Servers" and click "+ CREATE SERVER".

Fill in:

  • Server Name: My API Hub
  • Description: A collection of useful public APIs accessible through MCP

Step 9: Server Created

After creation, you'll see your server's main page with connection information.

Note the unique URLs generated for your server:

  • SSE Endpoint: For real-time connections
  • Streamable HTTP: For standard HTTP requests

Installing Toolkits and Creating Tool Instances

Step 10: Install Toolkits to Your Server

Click on the "Toolkits" tab to manage which toolkits are available in your server.

Your toolkit appears in the "Available" section. Click the ">>" button to install it.

Step 11: Create Tool Instances

Navigate to the "Tools" tab to see available tools and create instances.

Click the ">>" button next to each tool to create an instance. The system automatically:

  • Creates a tool instance with the same name as the tool
  • Makes it available as an MCP function
  • Discovers all parameters from the tool templates

Parameter Configuration

Step 12: Configure Parameter Bindings

Navigate to the "Bindings" tab to configure how parameters are resolved.

The system automatically discovered:

  • query - From the Advice Generator tool
  • base_currency - From the Currency Exchange Rates tool

Three-Tier Parameter Resolution

For each parameter, you can choose:

  1. Instance Level (Fixed Value)
    • Set a fixed value for this specific instance
    • Can include variables like {{api_key}}
  2. Server Level (Server Global)
    • Use a value from server globals
    • Shared across all instances in the server
    • Can be marked as secret (encrypted)
  3. Exposed (LLM Provided)
    • Leave empty to require the LLM to provide the value
    • These become function parameters in the MCP protocol

In our example, we left both parameters as "exposed" so the LLM can provide different values each time.

Testing Your Tools

Step 13: Test Tool Execution

Navigate back to your toolkit and click "TEST" on any tool.

The test page shows the request configuration. Click "EXECUTE TEST" to run the API call.

The test shows:

  • HTTP status code (200 OK)
  • Response body with the Chuck Norris joke
  • Response headers
  • cURL command for debugging

Connecting to Your MCP Server

Step 14: Get Connection Information

Return to your server's main tab to see connection details.

Your server provides two connection methods:

Connection Type URL Format Best For Protocol
SSE Endpoint https://[uuid].saramcp.com Real-time, streaming connections Server-Sent Events
Streamable HTTP https://saramcp.com/s/[uuid] Standard request/response JSON-RPC 2.0

Step 15: Connect with Claude or Other LLMs

To connect your server to Claude Desktop or Claude Code, use the provided connection URL. The server will:

  1. Respond to MCP protocol handshakes
  2. List all configured tool instances
  3. Accept tool calls with parameters
  4. Execute HTTP requests and return results

Key Concepts Summary

Core Components

Component Description Key Features
Toolkits Collections of related HTTP API tools
  • Reusable across multiple servers
  • Can be public (shared) or private
Tools HTTP request templates with typed parameters
  • Support GET, POST, PUT, DELETE, PATCH methods
  • Parameters use {{type:name}} syntax
Servers MCP endpoints with unique URLs
  • Import toolkits to make tools available
  • Configure access control (public/organization/private)
Tool Instances Configured instances of tools within a server
  • Each instance has a unique name (MCP function name)
  • Can override parameter bindings

Security Features

  • Secrets Management: Server globals can be marked as secrets
  • Encryption: AES-256-GCM encryption for sensitive data
  • Access Control: Three-tier access levels (public/organization/private)
  • OAuth 2.0: Standard authentication flow for protected servers

Best Practices

1. Organize Tools into Logical Toolkits

  • Group related APIs together
  • Use descriptive names and descriptions
  • Consider making useful toolkits public

2. Use Typed Parameters

  • Choose appropriate types for validation
  • Use descriptive parameter names
  • Document parameter purposes in tool descriptions

3. Configure Parameters Thoughtfully

  • Use instance-level for environment-specific values
  • Use server-level for shared configuration
  • Use exposed for dynamic, user-provided values

4. Test Before Deployment

  • Use the test interface to verify tools work
  • Check response formats and error handling
  • Save working configurations

5. Secure Sensitive Data

  • Mark API keys and tokens as secrets
  • Use server globals for shared credentials
  • Never expose sensitive data in public toolkits
